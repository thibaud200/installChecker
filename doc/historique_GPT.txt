# Vision du projet
## Mission

Le projet a pour objectif de créer un gestionnaire intelligent d'installateurs Windows capable d'analyser, 
d'identifier, de classifier et de maintenir une bibliothèque de plusieurs milliers, voire plusieurs centaines de milliers, 
d'installateurs.
Ce projet ne doit pas être considéré comme un simple détecteur de doublons.
Sa vocation est de devenir une plateforme d'analyse des installateurs Windows permettant de comprendre 
ce que représente chaque fichier, son origine, sa version, sa relation avec d'autres installateurs et son état 
par rapport aux versions actuellement disponibles.

## Vision long terme

À terme, le logiciel devra être capable de répondre automatiquement à des questions telles que :

Quel logiciel représente cet installateur ?
Quelle est sa version exacte ?
Qui est son éditeur ?
Est-il signé numériquement ?
Son certificat est-il valide ?
Existe-t-il plusieurs copies identiques ?
Existe-t-il plusieurs versions du même logiciel ?
Quelle version dois-je conserver ?
Quelle est la dernière version disponible officiellement ?
Où puis-je la télécharger ?
Ce logiciel existe-t-il déjà dans WinGet ?
Peut-on préparer une contribution WinGet ?

Le logiciel doit devenir un moteur de connaissance des installateurs Windows plutôt qu'un simple explorateur de fichiers.


# Philosophie

Le projet doit privilégier :

l'exactitude des informations ;
la fiabilité des analyses ;
la reproductibilité des résultats ;
la transparence des décisions ;
les performances ;
la modularité ;
l'évolutivité.

Chaque décision prise par le logiciel doit pouvoir être expliquée.

Le logiciel ne doit jamais agir comme une "boîte noire".

Lorsqu'une identification est réalisée, il doit être possible d'expliquer :

quelles informations ont été utilisées ;
quelles règles ont été appliquées ;
quel niveau de confiance a été obtenu.

## Ce que le projet n'est pas

Le projet n'est pas :

un antivirus ;
un moteur d'analyse de malware ;
un simple calculateur de hash ;
un simple détecteur de doublons ;
un gestionnaire de paquets ;
un explorateur de fichiers ;
un moteur de recherche Internet.

Ces fonctionnalités peuvent être utilisées comme briques mais ne constituent pas l'objectif principal.

# Public cible
Le logiciel est destiné notamment :

aux utilisateurs conservant une importante bibliothèque d'installateurs ;
aux administrateurs système ;
aux équipes IT ;
aux collectionneurs de logiciels ;
aux contributeurs WinGet ;
aux développeurs souhaitant maintenir un dépôt d'installateurs.

# Principes fondamentaux

Le projet repose sur plusieurs principes qui ne doivent jamais être remis en cause sans une Architecture Decision Record (ADR).

La donnée avant le fichier
Un installateur n'est pas seulement un fichier.
Il représente une version d'un logiciel.
Toutes les analyses doivent donc être orientées logiciel plutôt que fichier.
Les décisions doivent être explicables
Aucune décision ne doit être opaque.
Chaque résultat doit pouvoir être justifié.

Exemple :

identification grâce au certificat ;
confirmation par le ProductName ;
validation par le hash ;
score de confiance obtenu.
Les performances sont une fonctionnalité

La rapidité d'exécution fait partie des exigences fonctionnelles.

Une optimisation ne sera retenue que si elle est mesurée objectivement.

Les performances devront être évaluées par benchmark.

# Ambition du projet

L'objectif n'est pas de concurrencer un simple détecteur de doublons.
L'ambition est de construire une plateforme capable de devenir une référence pour la gestion intelligente des installateurs Windows.
Chaque fonctionnalité devra être pensée dans cette optique.
Lorsque plusieurs solutions sont envisageables, celle qui offre le meilleur compromis entre performances, évolutivité, maintenabilité et précision devra être privilégiée.

# Architecture

## L'architecture doit rester modulaire

Chaque composant devra être indépendant.
Le remplacement d'un module ne devra pas nécessiter la modification du reste du système.
Toutes les communications devront passer par des interfaces clairement définies.

## L'évolutivité est prioritaire

Le projet est conçu pour évoluer pendant plusieurs années.

L'ajout :

d'un nouveau type d'installateur ;
d'un nouveau catalogue ;
d'un nouveau moteur d'identification ;
d'une nouvelle source de téléchargement ;

ne devra pas nécessiter une réécriture du cœur du logiciel.

Les dépendances doivent être justifiées

Toute dépendance externe devra répondre aux critères suivants :

apporte une réelle valeur ;
est activement maintenue ;
possède une licence compatible ;
présente une stabilité suffisante ;
offre un gain significatif par rapport à une implémentation interne.
Les sources officielles sont prioritaires

Lorsqu'une information est disponible via plusieurs sources, la priorité est donnée :

aux métadonnées du fichier ;
à la signature numérique ;
aux catalogues officiels (WinGet, GitHub Releases, etc.) ;
aux sites officiels des éditeurs.

Les sources non officielles ne doivent être utilisées qu'en dernier recours.

## Vision de l'architecture

Le logiciel devra être conçu comme une plateforme composée de modules indépendants.
Le cœur du projet ne devra jamais dépendre d'un catalogue particulier.
Toutes les intégrations externes devront être réalisées au travers de connecteurs remplaçables.

Le moteur d'analyse devra être indépendant :

de la base de données ;
de l'interface graphique ;
de la ligne de commande ;
des catalogues externes.

Cette indépendance constitue un objectif majeur du projet.

## Vision de la qualité

La qualité du logiciel est aussi importante que ses fonctionnalités.
Chaque évolution devra préserver :

la lisibilité ;
la maintenabilité ;
la testabilité ;
les performances ;
la robustesse.

Toute dette technique devra être documentée.
Les choix d'architecture devront être motivés.
Les optimisations devront être mesurées.

1. La critique du pipeline actuel : Validée à 100 %
Votre distinction entre pipeline conceptuel (la suite logique d'idées) et pipeline physique (la réalité des lectures sur le disque) est le point le plus fort de votre analyse.

Le piège des lectures multiples : Ouvrir, lire et fermer un fichier de 50 Mo quatre fois de suite pour extraire les métadonnées, le PE, le hash et la signature est un non-sens absolu en termes de performance. Votre approche (un worker qui ouvre le fichier une seule fois et passe le flux d'octets ou le descripteur à tous les analyseurs locaux) divise immédiatement par 4 le goulet d'étranglement des I/O.

Le découplage des rythmes : Isoler la phase réseau (asynchrone, soumise aux rate limits et à la latence) du scan local est indispensable. 2. Analyse des architectures alternatives & Recommandation
Votre stratégie A → B (démarrer par le monolithe séquentiel pour livrer la Phase 1, puis migrer vers le modèle Producteur/Consommateur borné) est la meilleure approche possible.

Pourquoi l'Architecture B est le bon choix : L'utilisation de System.Threading.Channels en .NET est précisément conçue pour ce cas d'usage. Elle permet de gérer naturellement la backpressure (éviter que le scanner de fichiers ne sature la mémoire en listant 500 000 chemins alors que les workers CPU/Disque sont encore en train de traiter les premiers). De plus, l'écrivain SQLite unique élimine d'office les problèmes de verrouillage de base (database is locked).

Le rejet de l'Architecture C et de Rust : Très pertinent. Concevoir un système de plugins dynamiques (AssemblyLoadContext) pour un outil mono-utilisateur sans écosystème tiers est le meilleur moyen de passer 6 mois à coder de la plomberie plutôt que de la valeur métier. Quant à Rust, l'interopérabilité avec les API COM Windows (msi.dll) et les API natives comme WinVerifyTrust est tellement plus lourde qu'en C# que le gain CPU théorique (balayé par le fait que le goulot est le disque) se transformerait en dette de développement.

3. Focus sur le "Vrai problème difficile" (§1.4)
Vous avez mis le doigt sur le point faible des documents initiaux : la résolution d'identité.
Déterminer que setup.exe, Firefox Setup 128.0.1.exe et un binaire anonyme extrait d'un dossier temporaire pointent vers le même "Logiciel" est un enfer d'heuristiques. C'est ici que l'architecture B brille : en déchargeant la plomberie dans des channels et une base SQLite propre, vous dégagez du temps pour coder ce moteur de scoring.

La stratégie de pré-filtrage par hash partiel
Votre idée de faire Taille → Hash partiel (xxHash3) → SHA-256 complet est excellente pour la déduplication pure. Cependant, gardez en tête que pour la résolution d'identité, deux installateurs de versions différentes d'un même logiciel (ex: Firefox 127 vs Firefox 128) auront des tailles et des hashs 100% différents. Le hash partiel vous sauvera des I/O sur les doublons stricts, mais le moteur d'identité devra quand même aller lire le VersionInfo ou les ressources PE de presque tous les fichiers uniques.
